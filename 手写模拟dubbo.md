### 手写模拟dubbo
#### 项目目录
```
api ----- 模拟api-service，只提供接口
producer ----- 模拟producer-service，服务提供方
consumer ----- 模拟consumer-service，服务调用方
framework ----- 模拟dubbo框架的代码
```
#### 思路推演
怎么实现像dubbo那样, consumer像调用本地方法一样, 调到producer？
```
a. 生产者要和消费者之间通信, 怎么通信呢?
    1. 可以使用tomcat,然后将消费者请求转发到servlet去处理
    2. 也可以使用netty来连接
    这写不同的通信方式,我们可以抽象为协议
b. 首先生产者要将自己的服务信息注册到注册中心上去. 这样消费者才能找到.
    注册信息包括:
    服务的接口类型, 服务的具体实现类, 服务所在ip, 服务对于的端口, 使用的协议
    这些可以抽象成URL.
c. 注册中心选择
   共享内存的思想, 需要消费者和生产者能拿到相同的数据. 同时如果服务有变动,能通知消费者.
   比如:redis的发布订阅, zk的监听机制, nacos的推送等.
d. 消费者调用逻辑
   消费者根据接口生成代理对象,然后去拉取对应的URL.这里还可以扩展:
   比如:
   服务容错: 生产者有三个节点, 其中一个挂了, 可以去其他节点重试
   负载均衡: 根据策略,选择不同的生产者节点去调用
   代理方法中选到对应的生产者节点,然后将对应的信息传过去:
   接口类型, 方法名, 参数类型列表(有的服务有重载), 参数列表
 e. 生产者接收到信息,再通过反射去执行具体服务实现类的对应方法.返回结果  
```
